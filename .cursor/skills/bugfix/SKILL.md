# BUG 修复 Skill

## 目标

引导用户收集完整的报错信息和上下文，定位根本原因，修复 BUG，并生成规范化的修复报告供日后复盘。

---

## 核心理念

修 BUG 就三件事，缺一不可：

1. **报错信息** — 系统告诉你哪里出了错（错误日志、控制台报错、HTTP 状态码）
2. **报错上下文** — 你做了什么才触发的错（操作步骤、输入数据、环境状态）
3. **报错原因** — 为什么会出错（代码逻辑、数据问题、配置缺失）

前两个由用户提供，第三个由 AI 分析得出。**没有 1 和 2，不要急着猜 3。**

### Step 1: 收集报错信息（引导用户）

**向用户提问，收集第一要素：报错信息。**

用以下模板引导用户：

```
【收集报错信息】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请提供以下信息（能给多少给多少）：

1️⃣ 错误提示：你看到了什么报错？
   - 页面上显示了什么错误？
   - 浏览器控制台（F12 → Console）有没有红色报错？
   - 终端/命令行有没有报错信息？

2️⃣ 请把报错信息完整复制给我（截图也行）

💡 提示：报错信息越完整，定位越快。
   如果不确定哪里有报错，先打开浏览器 F12 看 Console。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**收到后，提取并记录：**

| 项目 | 内容 |
|------|------|
| **错误类型** | TypeError / 404 / 500 / SyntaxError / ... |
| **错误消息** | 完整的错误文本 |
| **出错位置** | 文件名:行号（如果报错信息里有） |
| **调用栈** | Traceback / Stack trace（如果有） |

### Step 2: 收集报错上下文（引导用户）

**向用户提问，收集第二要素：触发报错的操作和环境。**

```
【收集报错上下文】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
谢谢！现在我需要知道报错是怎么触发的：

3️⃣ 操作步骤：你做了什么操作之后出现的报错？
   - 例如："我点了登录按钮" / "我运行了 npm run dev"
   - 越详细越好，按顺序描述

4️⃣ 之前是正常的吗？
   - 这个功能之前能用吗？
   - 如果之前能用，最近改了什么？

5️⃣ 环境信息（可选但有帮助）：
   - 用的什么浏览器？
   - 前后端服务都启动了吗？
   - 数据库连上了吗？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**收到后，提取并记录：**

| 项目 | 内容 |
|------|------|
| **触发操作** | 用户做了什么触发的报错（操作步骤） |
| **复现条件** | 每次都出现，还是偶尔出现？ |
| **变更历史** | 之前正常吗？最近改了什么？ |
| **环境状态** | 服务是否启动、数据库是否连接、依赖是否安装 |

### Step 3: 分析报错原因

**1 和 2 都收集齐了，才开始分析 3。**

#### 3.1 问题分类

根据报错信息和上下文，判断问题属于哪一类：

| 类型 | 典型报错 | 排查方向 |
|------|----------|----------|
| **前端 UI** | 样式错乱、元素不显示 | CSS、组件模板、条件渲染 |
| **前端逻辑** | TypeError、undefined | Vue 组件、响应式数据、事件处理 |
| **API 调用** | 404、Network Error | 接口地址、请求参数、CORS、代理配置 |
| **后端接口** | 500、ValidationError | 路由、Service、请求体校验 |
| **数据库** | OperationalError | ORM、SQL、连接配置、迁移 |
| **配置/环境** | ModuleNotFoundError | .env、依赖、端口冲突 |

#### 3.2 定位根本原因

根据报错信息中的文件名和行号，直接检查相关代码：

| 报错线索 | 检查文件 |
|----------|----------|
| 报错信息里有明确的文件名:行号 | 直接打开该文件该行 |
| 报错在前端页面 | `frontend/src/views/Xxx.vue` |
| 报错在前端 API 调用 | `frontend/src/api/xxx.ts` |
| 报错在后端路由 | `backend/src/api/routes/xxx.py` |
| 报错在后端服务 | `backend/src/services/xxx.py` |
| 报错在数据库操作 | `backend/src/db/models.py` |
| 报错在配置 | `.env`、`config.toml`、`package.json` |

#### 3.3 输出分析结论

```markdown
## 报错原因

**根本原因**：[一句话说明为什么报错]

**出错位置**：`[文件路径:行号]`

**问题代码**：
```[语言]
[出问题的代码]
```

**为什么出错**：[技术层面的原因解释]
```

## 注意事项

1. **先问再查** — 不要跳过 Step 1 和 Step 2 直接猜原因，信息不全时主动追问
2. **保留原话** — 用户描述必须原样保留在报告中
3. **记录文件名** — 报告内容里必须包含出错文件的路径和行号
4. **最小改动** — 只修复 BUG，不要顺便重构或"优化"
5. **给操作指令** — 修完之后必须告诉用户下一步做什么来验证
6. **引导而非包办** — 引导用户学会看报错、找上下文，而不是替用户做所有事
